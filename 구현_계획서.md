## 파일 구조

기존 main.py 단일 파일 구조를 FastAPI APIRouter 기반으로 분리 완료.
기존 28개 엔드포인트 전부 routers/로 이관 완료. main.py는 진입점만 담당.
router 15개 파일, 총 4,640줄.

### main.py 구조

```python
import os
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from routers import (
    auth, signup, companies, rfqs, matching, orders,
    drawings, quotes, reviews, notifications, admin, catalog, legacy
)

app = FastAPI()
app.add_middleware(CORSMiddleware, allow_origins=["*"],
                   allow_credentials=True, allow_methods=["*"], allow_headers=["*"])

app.include_router(legacy.router)
app.include_router(signup.router)
app.include_router(auth.router)
app.include_router(companies.router)
app.include_router(rfqs.router)
app.include_router(matching.router)
app.include_router(orders.router)
app.include_router(drawings.router)
app.include_router(quotes.router)
app.include_router(reviews.router)
app.include_router(notifications.router)
app.include_router(admin.router)
app.include_router(catalog.router)
```

### 공통 모듈

```
routers/deps.py               ← DB engine, SCHEMA 변수, 인증 의존성, 유틸 함수
```

deps.py 포함 내용:
- `engine`, `SessionLocal`, `SCHEMA` — DB 접속
- `_hash_password(password)` — PBKDF2 해싱 (signup, admin 공용)
- `_verify_password(password, stored_hash)` — salt 분리 후 해시 비교
- `_check_onboarding(conn, company_id)` — 온보딩 자동 전환 (companies, matching 공용)
- `_refresh_mv(conn)` — MV REFRESH (companies, reviews, orders 공용)
- `_create_notification(conn, recipient_type, recipient_id, event_type, title, message, ref_id, ref_type)` — 알림 생성 헬퍼

인증 의존성:
- `get_current_user(token)` — JWT 디코딩 → `{id, login_id, role}` 반환
- `get_current_admin(token)` — admin 전용 의존성

### JWT 설계

```python
# JWT payload
{
    "sub": "<uuid>",              # buyer_id 또는 company_id 또는 admin_id
    "login_id": "hong_buyer",
    "role": "buyer" | "supplier" | "admin",
    "exp": 1234567890
}
```

### 로그인 ID: login_id (아이디)

이메일이 아닌 **login_id를 로그인 ID로 사용**한다. 이메일은 연락용으로만 사용하며 중복 가능.

- `buyers.login_id` — `citext NOT NULL UNIQUE` (전체 유일)
- `companies.login_id` — `citext NOT NULL UNIQUE` (전체 유일)
- `admins.login_id` — `citext NOT NULL UNIQUE` (전체 유일)

이메일 UNIQUE 제거:
- `buyers.email` — UNIQUE 제거 (NOT NULL 유지)
- `companies.main_email` — UNIQUE 제거 (NOT NULL 유지)

같은 이메일로 buyer + supplier 동시 가입 가능. 아이디만 다르면 됨.

### POST /signup 수정

signup request body에 `login_id` 필드 추가 필수:
- buyer: `{"login_id", "name", "email", "password"}` → buyers.login_id에 저장
- supplier: `{"login_id", "name", "email", "password"}` → companies.login_id에 저장

### 로그인 흐름

```
POST /api/login {"login_id": "hong_buyer", "password": "..."}
→ buyers에서 login_id 조회 → 있으면 password_hash 검증 → JWT (role=buyer) 발급
→ 없으면 companies에서 login_id 조회 → 있으면 검증 → JWT (role=supplier) 발급
→ 둘 다 없으면 404
```

role 파라미터 불필요. login_id가 어느 테이블에 있는지로 자동 판별.

관리자는 `POST /api/admin/login` 별도 엔드포인트 → admins 테이블만 조회.

### 소유권 검증

모든 인증 필요 API에서:
- JWT의 `sub`(buyer_id 또는 company_id)와 요청 대상 리소스의 소유자를 비교
- 불일치 시 403 Forbidden
- body에서 company_id를 받는 API (온보딩 등)는 JWT의 sub와 일치 여부 검증 필수

### 인증 불필요 API (public)

| API | 이유 |
|-----|------|
| `POST /signup` | 가입 |
| `POST /api/login` | 로그인 |
| `POST /api/admin/login` | 관리자 로그인 |
| `GET /api/processes`, `/api/material-categories`, `/api/materials`, `/api/equipment-categories`, `/api/equipment-models` | 카탈로그 조회 |
| `GET /companies`, `/companies/buyers`, `/companies/suppliers`, `GET /api/company/{id}` | 업체 목록/상세 조회 (public) |
| `GET /api/health` | 헬스체크 |
| `GET /`, `POST /predict`, `GET /db-test` | 레거시 유틸 |

나머지 전부 `Depends(get_current_user)` 적용.

모든 router가 `from routers.deps import engine, SCHEMA, get_current_user`로 공통 자원 참조.

## 구현 결과 (의존 관계 기준)

### Phase A: 기반 (다른 것들의 선행 조건) — 구현 완료

#### A-1. 로그인 (JWT) — 구현 완료
- **DDL 수정 (반영 완료)**:
  - `buyers`: `login_id citext NOT NULL UNIQUE` 추가, `email` UNIQUE 제거
  - `companies`: `login_id citext NOT NULL UNIQUE` 추가, `main_email` UNIQUE 제거
- **구현 완료 API**:
  - `POST /api/login` — login_id + password → buyers/companies 순차 조회 → `_verify_password` → JWT 발급
  - `GET /api/me` — 토큰에서 사용자 정보 반환
- **구현 완료 함수**: `_verify_password(password, stored_hash)` — deps.py에 배치
- **role 판별**: login_id가 어느 테이블에 있는지로 자동 판별. role 파라미터 불필요.
- **소유권 검증**: 모든 인증 API에서 JWT sub와 리소스 소유자 비교
- **적용 완료**: 인증 필요 엔드포인트에 `Depends(get_current_user)` 적용
- **의존**: 없음

#### A-2. 도면 파일 업로드 — 구현 완료
- **DDL**: drawings 테이블 기존 존재 (file_uri, file_sha256, original_filename)
- **구현 완료 API**:
  - `POST /api/drawings/upload` — multipart 파일 업로드 → 로컬/S3 저장 → drawings INSERT
- **저장**: 로컬 `uploads/` 폴더 (데모), 이후 S3 전환
- **의존**: A-1 (buyer 인증)

### Phase B: 매칭 후 흐름 — 구현 완료

#### B-1. RFQ 단건 조회 — 구현 완료
- **DDL**: 기존 존재
- **구현 완료 API**:
  - `GET /api/rfq/{rfq_id}` — rfqs + rfq_parts + rfq_part_processes JOIN 반환
- **의존**: 없음

#### B-2. 업체 단건 조회 — 구현 완료
- **DDL**: 기존 존재
- **구현 완료 API**:
  - `GET /api/company/{company_id}` — companies + sites + contacts + capabilities + equipment + reviews 종합 반환
- **인증**: public (인증 불필요). 업체 목록/상세 조회는 공개 API.
- **의존**: 없음

#### B-3. 매칭 이력 저장 — 구현 완료
- **DDL**: match_runs, match_candidates 기존 존재
- **수동 매칭**: 발주자가 RFQ 생성 + 보완 완료 후 [매칭 실행] 버튼을 눌러 매칭을 트리거한다.
  - 흐름: `POST /rfq` (RFQ 생성) → 보완 입력 → `POST /api/match-v2` (발주자가 버튼 클릭) → match_runs/match_candidates INSERT
  - VLM 연동 흐름: `POST /api/match-v2`가 도면 JSON을 받아 RFQ 생성 + 매칭 + 이력 저장을 한번에 처리 (기존 동작)
- **저장 데이터**: algorithm_version, mode, input_summary_jsonb, 후보별 hard_filter_pass/total_score/rank_no
- **알림**: 매칭 완료 시 buyer에게 match_completed, 매칭된 supplier에게 match_request 자동 발송
- **의존**: 없음

#### B-3b. 업체 수신 매칭 조회 — 구현 완료
- **DDL**: match_candidates 기존 존재
- **구현 완료 API**:
  - `GET /api/company/matches` — JWT의 company_id로 본인에게 온 매칭 요청 목록 조회 (match_candidates WHERE company_id = JWT.sub AND supplier_response = 'pending')
- **의존**: B-3

#### B-4. 업체 수락/거절 — 구현 완료
- **DDL 수정 (반영 완료)**: match_candidates에 컬럼 2개 추가:
  - `supplier_response text NOT NULL DEFAULT 'pending' CHECK (supplier_response IN ('pending','accepted','declined','expired'))`
  - `responded_at timestamptz`
- **구현 완료 API**:
  - `PUT /api/match-candidates/{match_run_id}/{company_id}/respond` — accept/decline + 타임스탬프
- **의존**: B-3 (매칭 이력이 먼저 저장되어야 수락/거절 가능)

#### B-5. 견적 비교 조회 — 구현 완료
- **DDL**: quote_responses 기존 존재
- **구현 완료 API**:
  - `GET /api/rfq/{rfq_id}/quotes` — 해당 RFQ의 모든 견적 + 업체 정보 JOIN 반환
- **의존**: 없음

#### B-6. 업체 선택 + 발주 확정 — 구현 완료
- **DDL**: orders 기존 존재
- **구현 완료 API**:
  - `POST /api/orders` — quote_id 받아서 order 생성 (status='contracting')
  - `GET /api/orders/{order_id}` — 발주 상세 조회
- **자동 처리**: rfqs.status → 'ordered', quote_responses.status → 'accepted', 나머지 quotes → 'rejected'
- **의존**: B-5 (견적이 있어야 발주 가능)

#### B-7. RFQ/Order 상태 전이 — 구현 완료
- **DDL 수정 (반영 완료)**: rfqs.status CHECK에 'quoted' 추가
- **구현 완료 API**:
  - `PUT /api/rfq/{rfq_id}/status` — 상태 변경
  - `PUT /api/orders/{order_id}/status` — 상태 변경
- **상태 머신**:
  - RFQ: open → quoted → ordered / closed / cancelled (closed: 기간 만료, cancelled: 발주자 취소). draft 미사용 (생성 시 바로 open).
  - Order: contracting → ordered → in_production → inspection → shipped → delivered → completed / cancelled / disputed
- **자동 전이**: 견적 도착 시 open→quoted (POST /api/quote 내부), 발주 시 quoted→ordered (POST /api/orders 내부)
- **수동 전이**: delivered→completed는 수동 전이 (리뷰 자동 전이 아님)
- **허용 범위**: in_production→cancelled 허용 (계획서 원안보다 넓게 적용)
- **POST /rfq 수정**: request body에 `drawing_id` 선택 필드 추가 (도면 업로드 후 연결용). rfqs.drawing_id에 저장.
- **의존**: B-6

### Phase C: 생산/납품 흐름 — 구현 완료

#### C-1. 가공 진행도 — 구현 완료

  - `GET /api/orders/{order_id}/jobs` — 발주 건의 진행도 조회
- **의존**: B-6 (발주 확정 후)

#### C-2. 납품 이미지 업로드 — 구현 완료

- **구현 완료 API**:
  - `POST /api/jobs/{job_id}/images` — multipart 이미지 업로드
  - `GET /api/jobs/{job_id}/images` — 이미지 목록 조회
- **의존**: C-1

#### C-3. 배송 정보 — 구현 완료
- **DDL 추가 완료**: `shipments (shipment_id, order_id, shipping_method, tracking_number, shipped_at, delivery_address, notes)`
- **구현 완료 API**:
  - `POST /api/orders/{order_id}/shipment` — 배송 정보 등록
  - `GET /api/orders/{order_id}/shipment` — 배송 정보 조회
- **의존**: C-1

#### C-4. 도면 접근 권한 제어 — 구현 완료
- **구현 완료**: `GET /api/drawings/{drawing_id}/download`
  - 매칭 단계: 스펙만 공개 (match_input 수준)
  - 발주 확정 후 (orders 존재): 도면 원본 다운로드 허용
- **로직**: order가 존재하고 해당 company_id가 요청자인지 확인
- **의존**: A-1 (인증), B-6 (발주)

### Phase D: 클라이언트 보완 + 용량 관리 — 구현 완료

#### D-1. 클라이언트 보완 요청 루프 — 구현 완료
- **DDL 수정 (반영 완료)**: `rfq_parts`에 `part_status` 컬럼 추가 (pending/matched/rejected/supplemented)
- **구현 완료**: rejected 부품에 대해 "이 필드를 채워주세요" 응답 + 클라이언트 재제출
- **구현 완료 API**:
  - `POST /api/rfq/{rfq_id}/supplement` — 누락 필드 보완 입력 (material, processes 등)
  - 내부에서 해당 rfq_parts UPDATE + 재매칭 트리거
- **의존**: 없음

#### D-2. 주간 용량 캘린더 — 구현 완료
- **DDL**: company_capacity_calendar 기존 존재 (planned_capacity_hours, booked_hours, available_hours 자동 계산)
- **주의**: `available_hours`는 `GENERATED ALWAYS AS` 컬럼이므로 INSERT/UPDATE 시 이 컬럼에 값을 넣으면 에러. planned/booked만 입력.
- **구현 완료 API**:
  - `POST /api/company/capacity` — 주간 용량 등록 (장비별, 공정별). planned_capacity_hours, booked_hours만 입력.
  - `GET /api/company/{company_id}/capacity?week=2026-W20` — 주간 가용 시간 조회
  - `PUT /api/company/capacity/{capacity_id}` — 용량 수정
- **의존**: 없음

#### D-3. 일단위 장비 스케줄 + 수주 on/off + 매칭 스코어 체계 — 구현 완료
- **DDL 추가 완료**: `equipment_daily_schedule` (일단위 장비 스케줄, available_hours 자동 계산)
- **DDL 수정 (반영 완료)**: `companies`에 `accepting_orders boolean NOT NULL DEFAULT true` 추가
- **MV 조건 추가**: `company_capability_summary` WHERE절에 `AND c.accepting_orders = true` 추가. 수주 off 업체는 매칭에서 제외.
- **구현 완료 API (3개 추가)**:
  - `POST /api/equipment/{equipment_id}/schedule` — 일별 스케줄 등록/수정 (단일 날짜 또는 from_date~to_date 범위 입력). UPSERT 방식. 장비 소유권 검증.
  - `GET /api/equipment/{equipment_id}/schedule?from=&to=` — 장비별 스케줄 조회. 기본 범위: 오늘 ~ 4주 후.
  - `GET /api/company/{company_id}/daily-schedule?from=&to=` — 업체 전체 장비 일단위 스케줄 조회. equipment JOIN으로 display_name 포함.
- **매칭 스코어 체계**: `_save_match_history` 내부에서 후보별 스코어를 산출하여 `match_candidates`에 저장한다.
  - `technical_score` = (18 - best_it_grade) / 18 — IT 등급 기반 기술력 점수
  - `availability_score` = 납기 가용성 (1.0: 충분 / 0.7: 빠듯 / 0.5: 데이터 없음 / 0.3: 불충족) — `_compute_availability_score`에서 `equipment_daily_schedule` 가용 시간과 `requested_delivery_date` 기반 산출
  - `quality_score` = avg_rating / 5.0 — 리뷰 평점 기반
  - `total_score` = technical(0.4) + availability(0.3) + quality(0.3) — 가중 합산
  - `rank_no` = total_score 내림차순 순위
- **매칭 결과 응답 보강**: pipeline 결과의 각 후보에 `technical_score`, `availability_score`, `quality_score`, `total_score`, `availability_info` 필드가 추가된다.
- **발주 시 자동 예약 제거**: `_auto_book_equipment_schedule` 제거됨. 장비 스케줄은 업체가 직접 관리한다.

### 상태 전이 권한 매트릭스

#### RFQ 상태
| 전이 | 트리거 | 실행 주체 |
|------|--------|----------|
| (생성 시 바로 open) | RFQ 생성 완료 | 시스템 (자동, draft 미사용) |
| open → quoted | 첫 견적 도착 | 시스템 (POST /api/quote 내부) |
| quoted → ordered | 발주 확정 | buyer |
| open/quoted → closed | 기간 만료 | 시스템 (스케줄러) |
| open/quoted → cancelled | 발주자 취소 | buyer |

#### Order 상태
| 전이 | 트리거 | 실행 주체 |
|------|--------|----------|
| contracting → ordered | 계약 완료 / 업체 수락 | buyer, supplier |
| ordered → in_production | 생산 시작 | supplier |
| in_production → inspection | 가공 완료 | supplier |
| inspection/shipped/delivered → completed | 검수 합격 (inspect pass) | buyer (inspect API) |
| inspection → in_production | 검수 불합격 → 재작업 | buyer (inspect API) |
| inspection → disputed | 검수 불합격 | buyer (inspect API) |
| shipped → delivered | 수령 확인 | buyer |
| delivered → completed | 수동 전이 (발주자 확인) | buyer |
| * → cancelled | 취소 | buyer/admin (contracting~in_production까지 허용) |
| * → disputed | 분쟁 제기 | buyer/supplier |

### 취소 시 연쇄 처리

#### RFQ 취소
1. `rfqs.status` → 'cancelled'
2. `match_candidates` WHERE rfq → `supplier_response` → 'expired'
3. `quote_responses` WHERE rfq → `status` → 'withdrawn'
4. 알림: 관련 업체에 'rfq_cancelled' 발송

#### Order 취소 (contracting~in_production 단계까지 허용)
1. `orders.status` → 'cancelled'
2. `manufacturing_jobs` WHERE order → `job_status` → 'cancelled'
3. `company_capacity_calendar` → `booked_hours` 환원: 미구현 (업체 수동 관리로 결정)
4. 알림: buyer/supplier에 'order_cancelled' 발송

### rejected 업체 재신청

`_check_onboarding`에서 rejected 상태도 재검증 가능하도록 변경:
```
현재: onboarding_status IN ('draft', 'submitted') → verified
변경: onboarding_status IN ('draft', 'submitted', 'rejected') → verified
```
관리자가 반려해도 업체가 문제를 수정하고 조건을 다시 충족하면 자동 복원.

## 미구현 사항

| 항목 | 사유 |
|------|------|
| delivered→completed 자동 전이 | 수동 전이로 결정. 리뷰 작성이 자동 전이를 트리거하지 않음. |
| booked_hours 자동 환원 | 업체 수동 관리로 결정. 주문 취소 시 capacity calendar / daily schedule의 booked_hours를 자동 차감하지 않음. |
| 발주 시 장비 자동 예약 | 제거됨 — `_auto_book_equipment_schedule` 삭제. 장비 스케줄은 업체가 직접 관리한다 (`POST /api/equipment/{id}/schedule`). |
| draft 상태 | 미사용. RFQ 생성 시 바로 open 상태로 진입. |
| 실시간 알림 (WebSocket) | 미구현. DB 폴링 방식으로 알림 조회. |
| 알림 만료/정리 | 미구현. 오래된 알림의 자동 삭제/아카이브 없음. |
